  <action>
Replace the stub templates/convert.md with a production template. Follow the structure of existing templates (prd.md for skill chaining, resolve.md for AskUserQuestion pattern).

The template must implement these steps:

**Step 1: Read Context**
- Read files_to_read (STATE, config, PHASE_FILES which includes prd.md)
- Read depth from config: node ralph-tools.cjs config-get depth
- Read .planning/pipeline/prd.md for full PRD content

**Step 2: Present Bead Format Gate**
- Use AskUserQuestion to present three format choices:
  1. **bd** -- Go beads via /ralph-tui-create-beads
  2. **br** -- Rust beads via /ralph-tui-create-beads-rust
  3. **prd.json** -- JSON format via /ralph-tui-create-json
- No config default -- always ask (locked decision from CONTEXT.md)

**Step 3: Detect Frontend Stories**
- Before invoking the bead creation skill, scan the PRD content for frontend-related keywords: "frontend", "UI", "interface", "component", "page", "view", "layout", "screen", "dashboard" (case-insensitive)
- For each user story that matches, note it as a frontend story
- After beads are created, these frontend stories' beads should have /frontend-design skill instruction injected into their acceptance criteria
- Detection is automatic, no user confirmation needed (locked decision)

**Step 4: Invoke Bead Creation Skill**
- Use the Skill tool to invoke the chosen skill
- Pass the full PRD content as input context
- Include depth instruction: "quick = fewer larger beads, comprehensive = more granular beads" with the actual depth value
- Wait for skill completion

**Step 5: Validate Bead Output**
- Check .beads/ directory exists
- Count .beads/*.md files using Glob
- If ZERO beads: write .planning/pipeline/convert.md with completed: false, present error gate:
  - **retry** -- re-invoke the skill
  - **edit PRD** -- tell user to edit .planning/pipeline/prd.md then re-run
  - **abort** -- fail the phase
  (locked decision: never proceed with empty .beads/)
- For each bead file, validate:
  - Has YAML frontmatter (between --- markers)
  - Has acceptance criteria section
  - Reject malformed beads with specific error per bead (locked decision)

**Step 6: Inject Frontend Design Skill**
- For beads matching frontend stories detected in Step 3:
  - Read the bead file
  - Prepend to acceptance criteria: "First, invoke /frontend-design for design guidance before implementing any UI."
  - Write the updated bead back

**Step 7: Write Completion File**
- Write .planning/pipeline/convert.md with frontmatter:
  completed: true
  format: {bd|br|prd.json}
  bead_count: {N}
  frontend_beads: {count of frontend-injected beads}
  depth: {depth value used}
- Body: list of all bead files with their story IDs

**Step 8: Return Completion**
- Return: ## PHASE COMPLETE

IMPORTANT implementation notes from research:
- The Skill tool IS available inside Task subagents (prd.md template confirms this pattern)
- If Skill tool invocation fails, fall back to describing the skill invocation as a natural language instruction
- Always quote bead paths in bash to handle names with spaces
- The template IS the Task subagent itself -- it runs in its own context window
