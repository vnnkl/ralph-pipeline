  <action>
Replace the stub templates/review.md with a production template. Follow the deepen.md template pattern closely -- same parallel agent dispatch, same P1/P2/P3 categorization. Key difference: review agents analyze git diff of code, not the PRD.

The template must implement these steps:

**Step 1: Read Context**
- Read files_to_read (STATE, config)
- Read .planning/pipeline/execute.md frontmatter to get pre_exec_commit
- Also try reading .claude/pipeline/pre-exec-commit.txt as fallback
- Aggregate bead results from .claude/pipeline/bead-results/*.md

**Step 2: Compute Review Diff**
- Get the pre-execution commit hash
- Run: git diff {pre_exec_commit}...HEAD
- If pre_exec_commit is "none" or missing, fall back to: git diff HEAD~{bead_count}...HEAD (best effort)
- Store the diff as REVIEW_DIFF to pass to each agent
- If diff is empty, warn but proceed (beads may not have committed)

**Step 3: Show Execution Summary**
- Display bead results summary: N passed, M failed, K blocked
- Display diff stats: N files changed, M insertions, K deletions
- This gives the user context before the review begins

**Step 4: Spawn Parallel Review Agents**
- Same pattern as deepen.md: 4 Task subagents with run_in_background=true
- Each agent reads its definition from ~/.claude/skills/compound-agents/review/
- Each agent receives the REVIEW_DIFF (not the PRD)
- Each writes findings to a distinct output file in .planning/pipeline/

**Agent 1 -- Security Sentinel:**
Prompt pattern: Read agent definition, then review the DIFF for security issues in actual code. Write to {{CWD}}/.planning/pipeline/review-security.md

**Agent 2 -- Architecture Strategist:**
Prompt pattern: Read agent definition, then review the DIFF for architectural concerns. Write to {{CWD}}/.planning/pipeline/review-architecture.md

**Agent 3 -- Code Simplicity Reviewer:**
Prompt pattern: Read agent definition, then review the DIFF for unnecessary complexity. Write to {{CWD}}/.planning/pipeline/review-simplicity.md

**Agent 4 -- Performance Oracle:**
Prompt pattern: Read agent definition, then review the DIFF for performance issues. Write to {{CWD}}/.planning/pipeline/review-performance.md

Each agent output file format (same as deepen):
---
agent: {agent-name}
completed: true
finding_count: {total}
p1_count: {N}
p2_count: {N}
p3_count: {N}
---
# {Category} Review
## P1 -- Critical
{findings or "None"}
## P2 -- Important
{findings or "None"}
## P3 -- Minor
{findings or "None"}

P1/P2/P3 thresholds (Claude's discretion per CONTEXT.md):
- P1 (Critical): Blocks merge -- security vulnerability, data loss risk, breaking change
- P2 (Important): Should fix before merge -- significant code quality issue, missing error handling
- P3 (Minor): Nice-to-have -- style, minor optimization, documentation

**Step 5: Collect and Present Findings**
- After all 4 agents complete, read their output files
- Parse frontmatter for counts, read body for details
- Deduplicate findings across agents (same issue found by multiple agents)
- Present combined report (same format as deepen.md Step 3)

**Step 6: Present Review Gate**
- Five options (locked decision from CONTEXT.md):
  1. **Fix P1s** -- Present P1 findings as a checklist. User applies fixes manually. No auto-fix agent (locked decision).
  2. **Fix P1+P2** -- Present P1 and P2 findings as a checklist. User applies fixes manually.
  3. **Skip** -- Accept all findings as-is, proceed without fixes
  4. **Re-run** -- Delete old review-*.md files, re-spawn all 4 agents fresh
  5. **Create PR** -- Create a draft PR via gh pr create --draft
  6. **Re-run bead X** -- Targeted re-execution of a specific bead (locked decision)

- For "Fix P1s" or "Fix P1+P2":
  - Present the findings as a numbered checklist
  - Wait for user to signal "done"
  - Then ask: "Re-run review to verify fixes, or proceed as-is?"

- For "Re-run":
  - Delete old review output files
  - Go back to Step 4

- For "Create PR":
  - Compute branch name from git branch --show-current
  - Run: gh pr create --draft --title "feat: {project_name}" --body "{summary with bead results and review findings}"
  - Report the PR URL
  - Then write completion file

- For "Re-run bead X":
  - Ask which bead to re-run (present list of beads)
  - Execute that single bead using the same claude -p pattern from execute template (with env -u CLAUDECODE)
  - Update its result file
  - After re-execution, offer to re-run review or proceed

**Step 7: Write Completion File**
- Write .planning/pipeline/review.md with frontmatter:
  completed: true
  findings_p1: {count}
  findings_p2: {count}
  findings_p3: {count}
  pr_created: {true|false}
  pr_url: {url or null}
- Body: final review findings summary
