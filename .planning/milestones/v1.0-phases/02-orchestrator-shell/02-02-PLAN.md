**Orchestrator logic (step by step):**

**Step 1: Load State**
```
Run: node ralph-tools.cjs init pipeline
Parse JSON output. Extract: current_phase, phase_name, status, mode, preflight_passed
```

**Step 2: Position Detection**
```
Run: node ralph-tools.cjs scan-phases
Parse JSON output. Find first incomplete phase.
Compare with current_phase from init.
If mismatch: log warning, use file scan position (per locked decision).
If pipeline_complete: show completion banner, stop.
```

**Step 3: Status Banner**
Display pipeline progress (from RESEARCH.md example):
```
## Pipeline Status

**Phase:** {id} of 9 ({name})
**Status:** Ready to dispatch
**Progress:** [##--------] {percent}%

Phase 1: Pre-flight -- {status}
Phase 2: Clarify -- {status}
...
```
Per locked decision: show status banner then auto-dispatch next incomplete phase.

**Step 4: Phase Dispatch**
```
Read template file: templates/{phase.template}
Fill variables using fillTemplate:
  CWD = current working directory
  PHASE_NAME = capitalized phase name
  PHASE_SLUG = phase.name
  PHASE_ID = phase.id
  STATE_PATH = .planning/STATE.md
  CONFIG_PATH = .planning/config.json
  PHASE_FILES = (empty for stubs, phase-specific later)

Dispatch as Task subagent with filled template content.
```

Per locked decision: orchestrator passes file paths only via `<files_to_read>` blocks -- never loads phase content into its own context.

**Step 5: Completion Verification (Dual)**
Per locked decision:
1. Check Task return message for `## PHASE COMPLETE` or `## PHASE FAILED` heading
2. Run: node ralph-tools.cjs scan-phases and check phase's completed status
3. If both agree on complete: proceed to gate
4. If completed: false or missing: present failure gate

**Step 6: User Gate**
Per locked decision: gates are context-dependent.

For successful completion:
```
Run: node ralph-tools.cjs excerpt .planning/pipeline/{phase}.md 10
```
Show summary + excerpt + gate options (from phase's gateOptions):
- **approve**: Run `node ralph-tools.cjs state set Status "Phase {id} complete"`, then suggest `/clear` for fresh context
- **redirect**: Spawn fresh Task subagent with original template + path to existing output + user feedback (per locked decision: fresh agent, not resume)
- **skip**: Mark phase as completed (write .planning/pipeline/{phase}.md with completed: true), advance
- **replan**: Return to planning mode (only shown for phases that support it)

For failure (after auto-retry):
Per locked decision: auto-retry once, then present failure gate.
- **retry**: Re-dispatch same phase
- **skip**: Mark skipped, advance
- **abort**: Stop pipeline, preserve state

**Step 7: /clear Boundary**
Per locked decisions:
- Manual mode: suggest `/clear first -- fresh context window` before next phase
- Auto mode: dispatch next phase as Task subagent (inherently context-isolated)
